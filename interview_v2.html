Q1.5追问：EWG 产品为什么从向量检索换成规则匹配？（真实叙事）
必背▼
真实叙事：POC 用向量检索 → 规模化时发现方向不对 → 切规则匹配。这是"技术决策有迭代"的故事，不是"一开始就想清楚"。
话术（背下来）：

"早期我们做了一个小规模的向量检索 POC——爬下来的 EWG 产品数据整理成 JSONL（字段是产品/品牌/使用方法/成分/安全指数），把其中的产品文本做 embedding 存进向量库，用 PRIMARY 成分做语义检索。
POC 跑起来后我发现两个问题：
第一 · 方法不对——向量检索对这个场景是错配的。
EWG 是结构化数据：品牌、成分、使用方法、安全指数。用户需求也是结构化的——'找含 retinol、不含 paraben、安全指数高' 的产品。这本质是布尔过滤 + 精确字段匹配问题，不是语义相似问题。

向量检索擅长的是捕捉语义等价（moisturizer ≈ hydrating cream），但对成分场景反而添乱——用户要 ubiquinone，向量会把 vitamin E、tocopherol 这些'语义相邻但用户没要'的抗氧化剂混进来。精确不了。

更关键的是 AVOID 成分必须硬过滤——用户过敏 paraben，含 paraben 的产品绝对不能出现在 top-k 里。向量检索的 top-k 按相似度排，AVOID 可能漏进去。护肤场景是安全红线，必须走确定性规则。
第二 · 规模化投入不划算。
POC 是几千条小规模验证，要扩到 135K 全量 embedding 还有时间和 API 成本要投。既然方向已经显示不对，就没必要再投入——直接切到确定性匹配方案。
最终架构：JSONL 不变，检索方式从向量改成规则——PRIMARY 成分命中 +10、SECONDARY +5、AVOID 硬过滤。代码可审计，每条推荐能追溯到具体规则。
医学文献还是用向量检索（Vertex AI RAG Corpus）——医学知识有真实的语义等价问题（痤疮 ≈ acne ≈ pimple inflammation），这是向量检索的主场。
最终原则：语义问题走向量（医学文献），精确匹配走规则（产品推荐）。两种各司其职。"
埋点："POC → 规模化发现方向不对" 是工程演进故事 · "结构化 vs 语义" 是技术判断 · "AVOID 硬过滤" 是安全意识

Q2最大技术挑战？怎么解决的？（带领导力原则）
必问▼
挑战1：幻觉控制
护肤场景幻觉是安全问题不是体验问题。在早期测试中，我发现LLM自由生成推荐时多次出现过推荐含过敏成分的产品。我意识到这个问题后，主动提出重新设计整个推荐层——当时队友觉得纯规则评分"太机械"，我用这些测试失败案例说服了团队（有骨气，敢于异议）。三层防护：Prompt层（system instruction硬约束"只基于SYSTEM CONTEXT回答"，视黄醇/AHA/BHA强制推荐SPF30+）；检索层（相似度低于阈值不注入）；推荐层（纯规则评分，不让LLM自由发挥）。
挑战2：Context Assembly延迟
用户画像+天气UV指数+日历事件+7天对话摘要四个数据源目前是串行读取，我识别出这是延迟的主要来源后（深入探究），主动加了内存TTL缓存（user_context_cache 120秒 / profile_cache 600秒）降低GCS读取频率——缓存命中时几乎零延迟。进一步优化方向是 asyncio.gather 并发读取，四个 IO 同时发出，理论上总耗时可以压缩到最慢那个 IO 的时间。asyncio.gather 是规划中的优化，没有实测数字。
如果面试官追问"这200ms代价怎么进一步优化"：优化分两步：第一步，Context Assembly的四个数据源改成asyncio.gather并发读取，这是改动最小、效果最显著的优化，理论上可以把串行总耗时压缩到最慢那个IO的时间；第二步，考虑Semantic Cache——对语义相似的query复用之前的检索结果，不重复调RAG。两个都是规划中的优化，没实测数字，说"预期""理论上"，不要说"已经降到 500ms"——没测过的数被追问会穿帮。

对比验证：同一问题跑两次，结果差异大说明模型输出不稳定，需要降低temperature或加更强约束

分级防护：Prompt层硬约束→检索层相似度阈值→推荐层纯规则评分，三层都失败才有风险

硬编码安全规则：视黄醇/AHA时强制推荐SPF30+，AVOID成分直接过滤，不经LLM判断

Q5Self-RAG？幻觉检测与自纠正？自适应路由？Agent状态机？（2026面试官想听的技术深度）
技术深度▼
面试官不再满足于"我用RAG检索+注入"，他们想听的是系统如何自我评估和纠错。

Self-RAG（自我反思RAG）：
普通RAG是一次性的：检索→注入→生成。Self-RAG在生成过程中加入反思步骤——模型在每个生成片段时同时判断：这个片段需要检索吗？检索到的内容相关吗？我的输出有幻觉吗？根据反思结果决定是否重新检索或修正输出。SkinMe的体现：置信度低于阈值不注入，宁可说"没找到"，就是简化版自我反思。

幻觉检测与自纠正：
检测：比较生成内容和检索source是否一致（Faithfulness检查）；检查输出是否引用了不存在的内容。
自纠正：检测到幻觉→重新检索（换query或换检索策略）；或者回退规则系统兜底。
SkinMe体现：发现LLM要推荐AVOID成分时，规则层直接拦截——这是硬编码的自纠正。

自适应路由（Adaptive Routing）：
不同问题适合不同检索策略。SkinMe的RoutingAgent就是自适应路由——根据intent类型决定走哪条链路（皮肤分析→Vertex AI RAG；产品推荐→EWG JSONL 确定性匹配）。更复杂的版本：根据query置信度动态决定走Dense、BM25还是直接LLM兜底。

Agent状态机设计：
把多Agent系统看成状态机：每个Agent是一个状态，状态转移有条件（检索成功→推荐；检索失败→重试/降级）。LangGraph就是在做这件事——有状态图，支持条件分支。SkinMe没有显式状态机，这是下一步要改的。

Q1Multi-Agent主流架构？Agent间怎么通信？
必问▼
🗣 嘴上怎么说：

四种架构模式：
— Pipeline（A→B→C线性）：每个Agent只负责一步，输出直接传给下一个。SkinMe用这个——Routing→Analysis→Recommendation→Response，固定串联。优点：简单可预测。缺点：不能回退不能分支。
— Router（路由分发）：一个Router Agent根据输入决定分发给哪个Worker Agent。SkinMe的路由Agent本质就是Router，只是用关键词规则而不是LLM做路由。
— Supervisor（主控调度）：一个Supervisor Agent持有所有Worker的描述，动态决定调用哪个Worker、以什么顺序。比Router更灵活——可以循环调用、组合调用。LangGraph 的典型用法。
— Peer-to-peer（对等协作）：没有中心调度，Agent之间直接通信协商。适合辩论/投票场景。实现复杂，SkinMe没用。

三种通信方式：
— 共享状态（SkinMe用这个）：所有Agent读写同一个dict——路由处理函数里analysis结果存进dict，recommendation从dict里读。优点：简单直接。缺点：Agent间有隐式依赖，改一个可能影响另一个。
— 消息传递：Agent之间通过事件/消息通信，异步解耦。LangGraph的event机制就是这个。优点：松耦合。缺点：调试复杂。
— 函数调用：Supervisor把Worker当Tool调用，传参数拿返回值。OpenAI Function Calling / Gemini Tool Use 就是这种模式。

SkinMe的选择及理由：
Pipeline + 共享状态，因为链路是确定性的4步串联，不需要动态调度。简单的问题用简单的架构——如果用Supervisor+消息传递，代码复杂度翻倍但效果一样。
追问：什么时候该从Pipeline升级到Supervisor？

"两个信号：
— 流程需要分支：'如果分析发现是严重皮肤病，走医生推荐链路而不是产品推荐'——Pipeline做不了条件分支
— 流程需要循环：'RAG召回不够好就换query再查'——Pipeline是单向的不能回头
出现这两个需求就该上状态机（LangGraph）或Supervisor。SkinMe目前不需要，但阿里的业务场景可能需要。"
对应代码：routing_agent.py（Router模式·classify_intent_fast）+ runner.py（Pipeline编排·run方法串联调用）+ route_and_process（共享状态·analysis结果传给recommend_products）
Q2Agent里有用到什么设计模式？（昨天答不上来）

我们事后复盘识别到了可以用的设计模式：

策略模式（Strategy Pattern）：RoutingAgent的5种intent各对应一个Strategy类，新增intent只需新增Strategy，不修改RoutingAgent本身。符合开闭原则，比大量if-else扩展性好。

观察者模式（Observer Pattern）：AnalysisAgent作为Subject，完成皮肤分析后发布事件，RecommendationAgent和DailyRoutineManager作为Observer自动收到并开始执行，互相独立。好处是新增Observer不需要修改AnalysisAgent。

Q3Skill太多占context太大？description相似加载错？
必问▼
🗣 嘴上怎么说：

问题一·Skill太多占context太大：
每个Skill的schema（名称+参数+描述）可能几百token。50个Skill就是上万token的context——还没开始对话就占满窗口了。

三种解法：
— 动态加载：根据用户当前意图，只注入最相关的2-3个Skill的schema，其余不加载。SkinMe里路由Agent先判断意图类型，只给后续Agent加载对应的工具描述。
— 两级路由：第一级Meta-Agent先判断大类（"这是产品问题还是皮肤问题？"），第二级在大类内部做细粒度Skill选择。减少每一层的候选数量。
— Description压缩：Skill描述只写"什么时候用我"和"我需要什么输入"，不写实现细节。面向LLM的描述要简洁精准，不是给人看的文档。

问题二·Description相似导致加载错误Skill：
比如"皮肤分析Skill"和"皮肤诊断Skill"描述很像，LLM可能选错。

三种解法：
— 加negative examples："我不处理XX场景"——比如皮肤分析Skill加"我不做产品推荐"，帮LLM排除干扰。
— 专门的Skill选择Agent：不靠LLM自己从description里选，而是有一个专门的Selector Agent做细粒度路由——它的唯一职责就是"这个请求应该调哪个Skill"。
— Few-shot典型case：description里加"典型触发场景：'我的痘痘反复发作' → 用我"，帮LLM通过例子区分。

SkinMe的做法：
SkinMe只有5种intent和4个Agent，不存在Skill爆炸问题——这是Pipeline架构的优势。但如果Agent数量增长到10+，上面的策略就必须用了。
追问：Tool Use / Function Calling 和 Skill 有什么关系？

"Tool/Function是执行层——LLM输出一个结构化调用指令，应用层执行并返回结果。
Skill是能力描述层——告诉LLM'你能做什么、什么时候用'。Skill的description就是LLM选择Tool的依据。
一个Skill可能对应多个Tool调用（比如'天气查询Skill'背后调了天气API+UV指数API+空气质量API三个Tool）。Skill是抽象，Tool是具体执行。"
Q4Agent设计最重要的？上下文工程（Context Engineering）完整版
必问▼
Agent设计最重要的两件事：

第一·职责边界清晰。
每个Agent只做一件事。SkinMe四个Agent各有明确职责：路由Agent只管意图分类不生成内容，分析Agent只管从RAG提取成分不管产品匹配，推荐Agent只管产品打分不管自然语言回复，编排器只管调度和最终回复。职责不清的后果：一个Agent又分析又推荐，出错了不知道是哪步出的问题。

第二·确定性优先。
路由是整条链路的第一步，错了后面全白做。所以路由用关键词规则不用LLM——100%可预测可审计。在不需要语义理解的环节坚决不用LLM。

Context Engineering（上下文工程）完整版：

Context Engineering比Prompt Engineering更上一层——不只是写好一条指令，而是系统性设计AI在整个任务过程中能看到什么信息、什么顺序看到、什么时候该注入什么时候不注入。

Prompt Engineering是语句级优化——这一句prompt怎么写。
Context Engineering是系统级设计——整个上下文窗口怎么组织。

四个核心原则：

1·重要内容放末尾（Lost in the Middle）
Stanford论文证实LLM对上下文中间部分注意力衰减。SkinMe里用户画像和安全约束（AVOID成分、SPF强制推荐）放在System Instruction末尾，不放中间。

2·无关内容比没有更有害
塞进去的无关信息会触发LLM联想产生幻觉。比如在护肤查询里注入用户的日历事件"下周有会议"，LLM可能联想"会议压力导致皮肤问题"编出一段话。宁可少给不要乱给。

3·历史用摘要替换全文
完整对话历史token成本爆炸且触发Lost in the Middle。SkinMe把7天历史压成约20词摘要（关键词规则抽取：过敏/敏感/皮肤问题/图片上传事件）。不是LLM摘要——规则抽取零延迟零成本。

4·动态组装，不固定注入
根据用户意图动态筛选注入什么：
— 问产品/成分 → 注入用户画像（肤质、过敏、敏感成分）——安全关键
— 有日历事件 → 注入日历（outdoor活动→高UV天提醒防晒）
— 有天气数据 → 注入天气（UV指数、温湿度）
— 有图片上传 → 注入完整context
— 简单问候（hi/hello/thanks且≤3词）→ 不注入任何context，节省token

为什么简单问候不注入：用户说"你好"时注入800 token的画像+天气+历史完全浪费——每次注入都有成本（token费+延迟+稀释有效信息）。

缓存和Context Engineering的关系：
Context组装本身有成本（读GCS拿画像、扫历史抽关键词、调天气API）。SkinMe用120秒TTL缓存组装好的context，同一用户多轮对话不重复组装。但天气请求绕过缓存——天气变化快，缓存会失真。
追问：如果上下文窗口不够放所有信息？

"四个策略按优先级：
1·压缩：历史用摘要替全文（已做）
2·筛选：只注入和当前query相关的context（已做）
3·分层：最重要的放system prompt（高注意力），次要的放user turn前文
4·外挂记忆：长期画像存向量库，用RAG按需召回而不是全量注入（规划中）"
Q5Agent并行时怎么保证数据一致性？
必问▼
🗣 嘴上怎么说：

"并发安全要在最接近数据的那一层保证，不能只靠上层逻辑控制。

三种保证方式：

1·乐观锁（适合读多写少）：
UPDATE SET balance=balance-100, version=version+1 WHERE version=当前version AND balance>=100
执行时才检查有没有人改过（通过version判断），失败就重试。不加锁不阻塞，适合冲突少的场景。
— 适用：用户画像更新（同一个用户不会同时从两个设备改画像）
— SkinMe对应：画像写入用GCS Object Versioning——写入时不锁，最后写入的覆盖前面的。因为画像更新频率极低（用户不会一直改肤质），乐观策略够了。

2·悲观锁（适合写冲突多）：
SELECT ... FOR UPDATE，读的时候就锁住行，其他事务只能等。保证串行执行。
— 适用：金融扣款（余额不能出错）
— SkinMe不需要：没有余额/库存这种关键写操作

3·Redis原子操作（适合高并发计数）：
DECRBY key 100 是原子操作，不需要锁。如果扣完返回负数，用Lua脚本回滚。
— 适用：限流计数器、库存扣减
— SkinMe不需要：没有计数器场景"
SkinMe 里的并发现状：

"SkinMe的Context Assembly目前是串行读取四个数据源（画像、天气、日历、聊天摘要）。计划改成并发读取——它们互相独立没有依赖关系，读操作天然线程安全，并发化不存在一致性问题。

实际存在的并发点是缓存层：多个请求可能同时读写SimpleCache。SkinMe用 threading.Lock 保护 get/set/invalidate——这是最简单但正确的方案。

另一个并发点：ProfileExtractor异步写GCS（post-response hook）。写GCS的同时下一个请求可能在读缓存——所以写完后必须立刻invalidate缓存，保证下一次读拿到最新数据。"
追问1：乐观锁和悲观锁什么时候选哪个？

"读多写少 → 乐观锁：不加锁不阻塞，冲突时重试。大部分Web场景。
写冲突多 → 悲观锁：直接锁住防止并发，牺牲性能保证正确。金融场景。
判断标准是冲突概率：冲突少用乐观（重试成本低），冲突多用悲观（重试太频繁还不如直接锁）。"
追问2：分布式环境下锁怎么做？

"单机用 threading.Lock / 数据库行锁。分布式用Redis分布式锁——SET lock_key owner_id NX EX 30，NX保证只有一个获取到，EX防死锁。
更严谨的用Redlock算法（多Redis节点投票），防止单Redis宕机导致锁丢失。
SkinMe是单Pod部署，threading.Lock够了。多Pod要换Redis锁。"
Q6Dense vs MoE？PG-MoE？ReAct/Function Calling/MCP区别？
高频▼
🗣 嘴上怎么说：

Dense vs MoE：
— Dense：每次推理所有参数都参与计算。参数量=计算量。
— MoE（Mixture of Experts）：多个Expert网络，每次只激活top-k个（通常2个），其余不算。总参数大但实际FLOPs低。GPT-4/Gemini/Mixtral都是MoE架构。
— 核心取舍：MoE用"更多参数但更少计算"换性能——同样计算预算下效果更好。代价是显存大（所有Expert都要加载）和负载均衡问题（某些Expert被过度调用）。

PG-MoE（我的研究项目）：
"这是我和MIT的同学合作的多模态活动识别项目。核心创新是在MoE的门控网络（Gating Network）里加入动作相位信息——从IMU时序数据提取'当前动作处于哪个阶段'（起始/高峰/结束），让门控感知时序上下文来选择Expert。

普通MoE的门控只看当前帧的特征向量，选Expert是静态的——不知道动作进行到哪了。
PG-MoE的门控看到的是'当前帧+动作阶段'，选Expert是动态的——动作起始阶段用擅长处理运动启动特征的Expert，高峰阶段用擅长处理快速运动的Expert。

视觉+IMU融合特征通过Phase-Gated门控分配给不同Expert，比普通MoE在动作识别benchmark上效果更好——尤其是在动作边界（切换点）的识别精度提升明显。"
ReAct / Function Calling / MCP 区别：

— ReAct：多步循环（Thought→Action→Observation→Thought...），LLM自主决策下一步做什么。适合开放复杂任务（"帮我规划旅行"）。
— Function Calling：单步调用，LLM输出结构化的函数调用指令（函数名+参数），应用层执行。适合固定流程。SkinMe用Gemini原生Tool calling（类似Function Calling）。
— MCP（Model Context Protocol）：Anthropic提出的标准化通信协议，类比USB——不同LLM和工具只要实现MCP接口就能互相插拔，不需要每对LLM-工具单独开发adapter。

三者关系：ReAct是推理范式（怎么思考），Function Calling是调用方式（怎么执行），MCP是连接协议（怎么对接）。可以组合——用ReAct范式+MCP协议连接外部工具+Function Calling方式调用。
RAG是被追问最久的部分。昨天面试官追问了chunk大小（噪声问题）、知识库更新、并行一致性。

为什么不整篇放进去（面试官追问的核心）：
整篇论文包含大量无关内容——作者介绍、参考文献、摘要、致谢，这些和具体皮肤科知识无关，但会被编码进向量里，成为噪声，降低检索相关性。切片能把这些无关噪声过滤掉，让检索结果更精确。

实验过语义分块：效果略好但计算成本高，短文本场景优势不明显，保留固定窗口。
面试官原话："适当切片可以把这些无关的东西过滤掉，让后面结果更精确。"两个原因都要说：①噪声干扰；②语义聚焦。

幻觉检测与自纠正（完整版·面试直接用）：

幻觉控制分两步：检测和纠正。很多系统只做了检测没做纠正——发现幻觉了直接报错，但没修复。

检测方法（三种）：
— Faithfulness检查：比较生成内容和检索source是否一致——生成内容里有没有source里不存在的事实？用NLI（自然语言推理）模型判断"生成内容能否从source推导出来"。RAGAS的faithfulness指标就是这个。
— 自一致性检查：同一个问题跑多次，如果答案差异大（temperature>0时），说明模型不确定，幻觉概率高。
— 规则校验：SkinMe的做法——推荐的成分必须在RAG结果里出现过，没出现过=幻觉，直接过滤。最简单但最可靠。

纠正方法（三种）：
— 重新检索：换query或换检索策略再查一次，用新context重新生成。
— 规则层拦截：SkinMe的做法——LLM要推荐AVOID成分时，规则层直接拦截不让输出。这是硬编码的自纠正，不依赖LLM自觉。
— 降级：检测到低置信度直接说"建议咨询皮肤科医生"。宁可不答也不答错——护肤场景的安全底线。

SkinMe的五层防御 = 检测+纠正的工程化：
RAG锚定（防止生成超出知识范围）→ 结构化输出（检测格式异常）→ 低temperature 0.3（减少随机性）→ Post-hoc校验（检测成分是否在RAG里）→ AVOID硬过滤（规则层纠正）

生产中幻觉检测的成本折中：
幻觉检测本身要调NLI模型或LLM判断——等于每个回复多一次模型调用，成本翻倍。折中方案：
— 高风险场景全量检测（金融/医疗推荐），低风险场景跳过
— 线上10%请求采样检测，幻觉率上升才全量开启

分级处理，不是全量上重检测。
Q4IVF vs HNSW？混合检索RRF融合？
高频▼
🗣 嘴上怎么说：

两种向量索引：
— HNSW（Hierarchical Navigable Small World）：图结构，多层跳表。recall高、查询快（~O(log N)），但内存占用大——全图都在内存里。适合千万级以下数据。POC阶段ChromaDB用的就是HNSW。
— IVF（Inverted File Index）：先用K-Means把向量聚成N个簇，查询时只搜最近几个簇。内存小（只存聚类中心），适合亿级数据。代价是需要预训练聚类，recall略低。Faiss的典型用法。

SkinMe的选择：
DermNet文献量级~几万篇，用Vertex AI RAG Corpus托管——底层用什么索引由Google管理，我们不直接控制。但知道这个量级HNSW最合适（内存兜得住，recall最高）。如果文献量级到百万+，要考虑IVF或混合索引。

混合检索 Dense + Sparse/BM25：
— Dense（向量）：捕捉语义等价——'moisturizer' ≈ 'hydrating cream'，适合医学文献检索（同义词多）
— Sparse/BM25（关键词）：对精确词更好——成分名'retinol'、产品名'CeraVe'，用关键词直接匹配比语义更准
— 为什么要混合：只用Dense会丢精确匹配（用户搜'CeraVe'但向量里最近的是别的品牌），只用BM25会丢语义等价（用户搜'补水'但文献写的是'hydration'）

RRF融合（Reciprocal Rank Fusion）：
两路检索各自排序后，按公式 score = 1/(rank+60) 合并排名。60是经验常数，让排名靠前的权重更大但不至于完全压倒另一路。免调参——不需要额外训练融合权重，直接用就行。
Vertex AI RAG内置了混合检索，权重平台自动调整，SkinMe不需要自己实现RRF。

SkinMe的双路架构和混合检索的关系：
— 医学文献用Vertex RAG的Dense检索（语义等价是主需求）
— EWG产品匹配用确定性字符串匹配（精确匹配是主需求）
这其实是在应用层做了"混合"——不是两种检索融合一个排名，而是两类数据各用最适合的检索方式。

Q3.5TCP状态机？CLOSE_WAIT堆积？TCP vs UDP选择？
补充▼
TCP状态机核心状态（面试追问常考）：
主动关闭方：ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT(2MSL) → CLOSED
被动关闭方：ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED

CLOSE_WAIT大量堆积的原因：服务器收到客户端FIN后没有及时调用close()，连接卡在CLOSE_WAIT。常见原因：代码里连接没有正确释放（忘记close/finally块没执行）。排查：netstat -an | grep CLOSE_WAIT

TIME_WAIT大量堆积：主动关闭方进入TIME_WAIT等2MSL。高并发短连接会堆积大量TIME_WAIT。解决：开启SO_REUSEADDR；或改用长连接Keep-Alive。

TCP vs UDP选择：
用TCP：需要可靠传输+顺序保证（HTTP、文件传输、数据库连接）
用UDP：对延迟敏感、允许少量丢包（视频通话、游戏、DNS查询、Audeate实时生理信号传输）
HTTP/3底层用QUIC（基于UDP），既有可靠性又有低延迟。
Q3.5TCP状态机？CLOSE_WAIT堆积？TCP vs UDP？
补充▼
TCP状态机：主动关闭方：ESTABLISHED→FIN_WAIT_1→FIN_WAIT_2→TIME_WAIT(2MSL)→CLOSED；被动关闭方：ESTABLISHED→CLOSE_WAIT→LAST_ACK→CLOSED

CLOSE_WAIT大量堆积：服务器没有及时调用close()，连接卡住。排查：netstat -an | grep CLOSE_WAIT，原因通常是代码里连接没有正确释放（忘记close/finally块）。

TIME_WAIT大量堆积：高并发短连接场景。解决：开启SO_REUSEADDR；或改用Keep-Alive长连接。

TCP vs UDP：用TCP：需要可靠传输（HTTP/文件/数据库）；用UDP：对延迟敏感允许丢包（视频通话/游戏/DNS/Audeate实时生理信号）。HTTP/3底层QUIC=UDP+可靠性。
Q4公钥私钥？常见加密算法？数据包封装？TTL？
高频▼

Q1进程线程区别？IO多路复用？Python GIL？
必问▼
进程：独立内存，崩溃不影响他人，开销大。线程：共享内存，开销小，需要锁处理竞争。最优线程数：CPU密集≈核心数；IO密集=核心数×(1+IO等待/计算时间)
epoll（IO多路复用）：一个线程监控多个fd，哪个就绪处理哪个。Nginx/Redis/FastAPI底层都用这个，高并发基础。
Python GIL：同一时刻只允许一个线程执行Python字节码，CPU密集多线程没用。IO等待时GIL释放→IO密集多线程有效。优化：CPU密集→multiprocessing；IO密集→asyncio（FastAPI用这个，绕过GIL）；数值计算→NumPy/PyTorch（底层C++释放GIL）
Q2死锁？僵尸进程？进程间通信？
高频▼
死锁四个必要条件：互斥/占有并等待/不可抢占/循环等待。解法：资源按固定顺序申请；tryLock超时放弃；InnoDB自动检测回滚。
僵尸进程：子进程已退出但父进程没有wait()回收，占用PID不占CPU。大量堆积耗尽PID。解法：父进程调用wait()；或init进程（PID=1）接管回收。Docker容器里pid=1的进程要特别注意。
进程间通信：管道（单向，父子进程间）/ 消息队列（结构化，异步）/ 共享内存（最快，需信号量同步）/ 信号量 / Socket（最通用，K8s里Pod间通信本质是Socket）

Q2MVCC？索引原理？索引失效？超扣怎么防？
必问▼
MVCC：让读不阻塞写，每行有trx_id+roll_pointer，读历史快照版本。REPEATABLE READ事务开始时生成Read View不变；READ COMMITTED每次SELECT生成新Read View。
B+树索引：数据全在叶子节点，叶子双向链表，树高3-4层，千万级3次IO找到。聚簇索引（主键）叶子存完整行；非聚簇索引叶子存主键值需回表；覆盖索引不需要回表性能最好。
索引失效：对索引列做函数（WHERE YEAR(time)=2024）/ 隐式类型转换 / LIKE左模糊（'%张'）/ OR有非索引列 / 联合索引不满足最左前缀。排查用EXPLAIN，type=ALL是全表扫描
超扣防止：乐观锁（WHERE version=当前version AND balance>=100）；悲观锁（SELECT...FOR UPDATE）；Redis DECRBY原子操作；金融核心：悲观锁+CHECK约束(balance>=0)双保险
Q3redo/undo/binlog？两阶段提交？
高频▼
redo log：InnoDB，记录"做了什么修改"，崩溃恢复（持久性），WAL先写日志
undo log：InnoDB，记录修改前数据，事务回滚（原子性）+MVCC历史版本
binlog：Server层，记录所有DDL/DML，主从复制+数据恢复
两阶段提交：redo log(prepare)→binlog→redo log commit，保证两者一致。

Q2PG-MoE是什么？LoRA微调是什么？
高频▼
PG-MoE（相位门控混合专家）：用于多模态人体活动识别。在门控网络里加入动作相位信息（从IMU时序提取），感知"当前动作哪个阶段"，动态选择对应Expert处理Qwen2.5-VL视觉特征+IMU传感器流的融合。UTD-MHAD数据集，7类高动态动作。

Q5手撕LRU（必考）
必考▼
结构：哈希表+双向链表。哈希表→O(1)找节点；链表→头部=最近用，尾部=最久没用，满了删尾。哨兵节点：head/tail是假节点，不存真实数据，避免边界判断。
get：找到→从当前位置摘掉→移到头部→返回val；找不到→-1
put：已存在→删旧节点；新建节点→加到头部；超容量→删尾部节点+从哈希表删
为什么双向链表：删节点需要知道前驱，单向链表删除O(n)。为什么哈希表：链表查找O(n)，哈希O(1)。get/put都是O(1)

核心幂等性设计——腾讯面试最大失分点

2. 扣款Agent执行前，先查询这个order_id是否已存在

4. 不存在→正常执行，写入order_id和结果
数据库给order_id字段加唯一索引，重复INSERT失败（唯一键冲突），捕获异常返回原结果。即使并发也能保证只有一个成功。

题1亚马逊货物盒子（长宽高）数据库索引精确搜索（面经真题）
面经真题▼
先澄清："是精确匹配（刚好10x20x30）还是范围查询（高度在20-30之间）？数据量级是多少？"

为什么数组不适合做Key：B+树索引对标量值高效，对数组类型支持有限，且数组中元素顺序不同（[10,20,30] vs [20,10,30]）即使代表相同尺寸也会被当作不同key。
方案1——排序后拼接字符串（精确匹配）：key = f"{min(l,w,h)}_{mid}_{max(l,w,h)}"，建唯一索引，精确匹配O(log n)。如果盒子旋转无关，相同尺寸不同顺序会生成相同key。

方案2——复合索引（范围查询）：对(length, width, height)三列建复合索引，满足最左前缀原则的范围查询高效。

方案3——Hash Key（高并发精确查找）：对标准化三元组计算哈希值，O(1)查找，但不支持范围查询，需处理哈希冲突。
推荐：先问清楚查询模式再选方案，这是以客户为中心的设计方式——了解实际使用场景再设计。

思路：哨兵头节点，比较两链表当前节点值，较小的接到结果链表，指针前移。时间O(m+n)，空间O(1)。

def mergeTwoLists(list1, list2):
  dummy = ListNode(0) # 哨兵节点，避免边界判断
  cur = dummy
  while list1 and list2:
    if list1.val <= list2.val:
      cur.next = list1
      list1 = list1.next # 注意：不是list2.next
    else:
      cur.next = list2
      list2 = list2.next
    cur = cur.next
  cur.next = list1 if list1 else list2
  return dummy.next

C5性能指标——FCP/LCP/CLS + script async/defer
高频▼
三大Core Web Vitals：
FCP（First Contentful Paint）：首次有内容绘制，衡量"页面是否在加载"。
LCP（Largest Contentful Paint）：视口内最大元素渲染完成，目标<2.5s，Google核心排名指标。LCP是动态追踪的——浏览器持续记录最大元素，用户首次交互时固化，之后不再变化。
CLS（Cumulative Layout Shift）：布局偏移累计值，目标<0.1。图片没设宽高、字体加载后撑开都会导致CLS高。

script标签 async vs defer：
普通script：遇到就停止HTML解析，阻塞渲染，应放body底部。
async：下载和解析并行，下载完立即执行，顺序不确定，适合独立脚本（统计）。
defer：下载和解析并行，HTML解析完才按顺序执行，能保证顺序，适合有依赖的脚本。

TypeScript扩展window类型：新建 global.d.ts，写 declare global { interface Window { myProp: string } }，这叫Declaration Merging，TS会合并声明。文件末尾必须加 export {} 让它成为模块。
② JS / ES6+
J1var/let/const——三个维度区分
必问▼
维度1 能否重新赋值：var/let可以，const不行。但const锁的是地址——const obj={}，obj.name='x' 不报错，obj={} 才报错。

维度2 作用域：var是函数作用域，无视花括号会泄漏；let/const是块级作用域，出了{}就消失。

维度3 变量提升：var声明提升到函数顶部，赋值不提升，访问得undefined不报错。let/const也提升但进入"暂时性死区（TDZ）"，声明前访问直接报错，更好排查问题。

现代代码原则：默认用const，需要改值用let，var基本不用。

==和===区别：==做类型转换再比；===严格相等，类型不同直接false。[] == false是true（[]→""→0，false→0），[] === []是false（引用不同）。
J2闭包——原理、用途、过期闭包陷阱
必问▼
定义：函数记住了自己被创建时的词法作用域，即使那个作用域已执行完毕。count 没有被垃圾回收，是因为内部函数持有对它的引用，只要函数还活着，count 就不会消失。

三个用途：数据私有化（外部无法直接访问count）；函数工厂（multiply(2)返回永远乘2的函数）；回调函数保留状态。

过期闭包（Stale Closure）：useEffect依赖数组写空[]，但内部用了外部变量，那个变量在闭包里被锁在初始值。
useEffect(() => { setInterval(() => setCount(count+1), 1000) }, []) ← count永远是0，结果是count始终=1。
修复：用函数式更新 setCount(prev => prev+1)，prev是React传入的最新值，不走闭包。

内存泄漏：闭包持有大数据的引用会阻止垃圾回收。定时器未清除、DOM引用未释放、全局变量都是常见原因。
J3事件循环——同步/微任务/宏任务执行顺序
必问▼
执行顺序：同步代码 → 微任务（Promise.then / async/await）→ 宏任务（setTimeout / setInterval）

经典题：
console.log('1') // 同步，输出1
setTimeout(()=>console.log('2'), 0) // 宏任务，排队
Promise.resolve().then(()=>console.log('3')) // 微任务，排队
console.log('4') // 同步，输出4
输出顺序：1 → 4 → 3 → 2。setTimeout(fn,0) 不是立即执行，是等同步和微任务都跑完再执行。

var在for循环的陷阱：
for(var i=0;i<3;i++){setTimeout(()=>console.log(i),0)} 输出 3 3 3，因为var共享同一个i，setTimeout执行时循环已跑完i=3。改用let就输出0 1 2，let每次循环创建独立的i。

原型链：每个对象有__proto__指向原型，一直到null。创建没有原型的对象：Object.create(null)。
J4Promise / async/await——串行vs并行
必问▼
Promise三种状态：pending → fulfilled（resolve，触发.then）/ rejected（reject，触发.catch）。状态不可逆。

async/await是Promise的语法糖：写起来像同步代码，await等Promise完成再继续，错误用try/catch捕获。SkinMe里调用Gemini API全用这个写法。

串行 vs 并行——面试高频：
串行（慢，2秒）：const a = await fetch('/a'); const b = await fetch('/b') — b等a完成才开始
并行（快，1秒）：const [a,b] = await Promise.all([fetch('/a'), fetch('/b')]) — 同时发出

Promise.all vs Promise.allSettled：Promise.all一个失败全部失败；Promise.allSettled等所有完成，每个结果有fulfilled/rejected状态，不会因一个失败崩掉。

手写防抖：function debounce(fn,delay){let t;return function(...args){clearTimeout(t);t=setTimeout(()=>fn.apply(this,args),delay)}}
测试方案：连续触发只执行一次；停止后delay时间内执行；this指向正确。

J6手写题——并发限制请求池（limit=3）/ 冒泡排序
高频▼
并发限制请求池：
function concurrentLimit(tasks, limit) {
  return new Promise(resolve => {
    const results = []; let index = 0, finished = 0
    function runNext() {
      if (index >= tasks.length) return
      const i = index++
      tasks[i]().then(res => {
        results[i] = res; finished++
        if (finished === tasks.length) resolve(results)
        else runNext()
      })
    }
    for (let i = 0; i < Math.min(limit, tasks.length); i++) runNext()
  })
}
核心：先启动limit个，每完成一个立即拉起下一个，不是分批。
冒泡排序：
function bubbleSort(arr) {
  for (let i = 0; i < arr.length-1; i++) {
    let swapped = false
    for (let j = 0; j < arr.length-1-i; j++) {
      if (arr[j] > arr[j+1]) { [arr[j],arr[j+1]]=[arr[j+1],arr[j]]; swapped=true }
    }
    if (!swapped) break // 优化：无交换说明已有序
  }
  return arr
}
O(n²)最坏，O(n)最好（加了swapped优化），空间O(1)原地排序。
③ React 框架
R1核心思想——数据驱动视图 / useState / useEffect
必问▼
React是什么：
Facebook开发的前端框架，用来构建用户界面。核心思路是数据驱动视图——你只管维护数据，界面自动跟着更新，不需要手动操作DOM。现在是国内外最主流的前端框架之一，SkinMe就是用React + Next.js做的。
🗣 "React是Facebook出的前端框架，核心思路是数据驱动视图，数据变了页面自动更新，不需要手动找到DOM元素去改它。我用React做了SkinMe的整个前端，包括流式对话界面、多模态输入、Agent状态可视化这些。"
数据与视图分离：你只管数据（state），UI自动跟着变，不需要手动操作DOM。setState → 组件重新渲染 → 新UI，这是React的核心循环。

setState是异步的：调用setCount后当前帧count不变，下次渲染才是新值。连续多次setState会被批处理，不会每次都触发渲染。

useEffect依赖数组：
useEffect(fn, []) — 只在挂载时跑一次
useEffect(fn, [id]) — id变了才重新跑
useEffect(fn) — 每次渲染都跑（一般不用）
return里写清理函数，组件销毁时执行（清除定时器、取消订阅）。

useMemo / useCallback：useMemo缓存计算结果，useCallback缓存函数本身，都是"依赖没变就不重新创建"，用在性能敏感的地方。

高频▼
useRef两个用途（React专属）：
1. 获取DOM节点：给某个HTML元素贴一个引用标签，之后在JS里可以直接拿到那个元素。
const ref = useRef(null)，<div ref={ref}>，通过ref.current访问真实DOM。
🗣 "useRef可以给某个元素贴一个引用标签，之后直接访问真实的DOM节点，比如自动聚焦输入框、控制视频播放这类需要直接操作元素的场景。"

2. 存不触发渲染的值：ref.current改变不会触发重渲染，适合存定时器ID、缓冲数据等。
🗣 "useRef还可以存一些不需要触发渲染的值。比如定时器ID，你只需要在清除时用到它，不需要每次改变都重新渲染。SkinMe里用ref缓冲SSE推来的token，攒一批再统一更新界面，不是每个token都setState，这样不卡。"

🗣 "useContext解决的是深层组件传数据麻烦的问题。不用一层一层传props，把数据放到Context里，任何层级的组件都可以直接取到。SkinMe里语言设置和登录状态放在Context里，每个组件直接读，不用从顶层一路传下来。"
R2bVue基础——是什么 / v-show vs v-if / ref vs reactive
高频▼
Vue是什么：前端框架，和React类似，都是数据驱动视图——数据变了页面自动更新。Vue有自己的模板语法（v-if/v-show这些指令），更接近传统HTML，国内大厂用得多。
🗣 "Vue和React核心思路一样，都是数据驱动视图。区别是Vue有模板语法，更接近传统HTML，国内大厂用得多。我主要用React，Vue是了解和能读懂的程度，基础概念都能回答。"
v-show vs v-if：v-show用display:none隐藏，DOM还在，切换快；v-if直接移除DOM，重新显示要重新创建，代价更大。
🗣 "v-show和v-if的区别是藏起来还是删掉。v-show只加display:none，元素还在，切换快，适合频繁显示隐藏。v-if是真的把元素从DOM里删掉，适合条件很少变的场景。"
ref vs reactive（Vue3响应式API）：ref包装单个值（数字/字符串/布尔），读取时要.value；reactive只包装对象，直接访问属性，不用.value。底层都是Proxy。
🗣 "Vue3的ref和reactive都是响应式API，区别是包装的数据类型不同。ref包装单个值，读取时要.value；reactive包装对象，直接访问属性。底层都是Proxy实现的，能监听到属性的增删改。注意这个Vue的ref和React的useRef完全不同——Vue的ref是响应式数据，React的useRef是存引用用的。"
vue2 vs vue3：vue3用Composition API，相关逻辑可以放在一起，不再按data/methods强制分开；响应式从Object.defineProperty改为Proxy，能监听数组和新增属性。
🗣 "vue2到vue3最重要的两个变化：一是Composition API，相关逻辑可以放在一起，不再强制分开；二是响应式从Object.defineProperty换成Proxy，vue2监听不到新增属性和数组下标变化，vue3用Proxy代理整个对象，这些都能拦截到了。"
R3Next.js渲染模式——SSR/SSG/ISR/CSR
必问▼
CSR（客户端渲染）：HTML空壳，逻辑全在浏览器跑。首屏慢，SEO差，适合不需要SEO的交互重页面，如数据可视化。
SSR（服务端渲染）：每次请求在服务器生成HTML。首屏快，SEO好，服务器压力大。适合内容实时变化的页面。
SSG（静态生成）：构建时提前生成HTML，所有人共享同一份文件，速度最快，适合内容不变的页面（文档/博客）。
ISR（增量静态再生成）：SSG升级版，设置revalidate时间，过期后自动重新生成，兼顾速度和新鲜度。

SkinMe里的混合使用：对话界面SSR保证实时，成分可视化CSR，Landing页面SSG。Next.js SSR让服务端直接返回渲染好的HTML，跳过客户端首次渲染，缩短FCP。

5. 服务端处理：Nginx接收→路由→业务逻辑→返回HTML
6. 浏览器接收响应：检查缓存头决定是否缓存
7. HTML解析：构建DOM树，遇到CSS暂停等样式，遇到普通script暂停等脚本
8. CSS解析：构建CSSOM树
9. Render Tree：DOM+CSSOM合并，过滤display:none的节点
10. Layout（重排）：计算每个节点的位置和大小
11. Paint（重绘）：把节点画成像素
12. Composite（合成）：多个图层合并输出到屏幕
13. TCP四次挥手：连接关闭（keep-alive可复用连接加载静态资源）

HTTP/HTTPS区别：HTTP明文传输；HTTPS=HTTP+TLS，加密传输防窃听，需要证书，端口443。
N2TCP三次握手/四次挥手——为什么是这个次数
高频▼
三次握手：SYN（我要连接）→ SYN+ACK（我收到了，我也要连接）→ ACK（我收到了）
为什么是三次：两次不够——服务端无法确认客户端能收到自己的回复；四次多余——第二步合并了确认和请求。三次是能确认双方收发都正常的最少次数。

四次挥手：FIN（我不发了）→ ACK（收到）→ FIN（我也不发了）→ ACK（收到）
为什么是四次：TCP是全双工，两个方向独立关闭。客户端说"我发完了"，服务端可能还有数据没发完，所以ACK和FIN必须分开，不能合并成三次。

TIME_WAIT状态：客户端发最后一个ACK后不能立即关闭，要等2MSL（报文最大生存时间），确保服务端收到ACK，防止旧连接的包影响新连接。

HTTP/1.1 vs HTTP/2：HTTP/1.1每个请求一个连接（keep-alive复用但串行）；HTTP/2多路复用，一个连接并行多个请求，头部压缩，性能大幅提升。
N3浏览器缓存——强缓存 vs 协商缓存
必问▼
强缓存：浏览器直接用本地缓存，不发请求。响应头 Cache-Control: max-age=3600 控制（旧版用Expires）。命中时状态码200（from disk cache）。

协商缓存：强缓存过期后，发请求问服务端"资源变了吗"。两种方式：
Last-Modified/If-Modified-Since：文件修改时间，精度到秒，1秒内多次修改识别不到。
ETag/If-None-Match：文件内容哈希，更精确，优先级高于Last-Modified。
命中协商缓存返回304 Not Modified，不返回响应体，节省带宽。

实际策略（SkinMe）：静态资源（JS/CSS）用强缓存+文件名哈希（main.abc123.js），内容变了文件名就变了；HTML不缓存，每次都拿最新入口。
N4同源策略 + 跨域——服务端收到请求吗？
必问▼
同源：协议+域名+端口三者完全一致。浏览器限制跨源JS读取响应，只限制读，不限制发。

跨域请求服务端收到吗？收到。浏览器会发请求，服务端处理并返回，但响应头没有 Access-Control-Allow-Origin 时，浏览器把响应拦截，JS拿不到。跨域限制是浏览器安全机制，不是网络层的。

CORS预检（Preflight）：PUT/DELETE/自定义Header等非简单请求会先发OPTIONS请求，服务端回复允许后再发真正的请求。

解决跨域：服务端设CORS响应头（最常见）；开发环境webpack/vite proxy转发；Next.js API Routes做BFF层（SkinMe用的，前端请求同源的Next.js，Next.js再请求FastAPI）；Nginx反向代理统一域名。
N5浏览器存储——Cookie/localStorage/sessionStorage
高频▼
Cookie：4KB上限，每次HTTP请求自动携带，可设HttpOnly（防XSS，JS无法读）、Secure（只HTTPS传输）、过期时间。主要用于身份认证。
localStorage：5MB，持久化到磁盘，不随tab/浏览器关闭销毁，需手动清除。同源共享，不随请求发送。
sessionStorage：同localStorage，但只在当前tab有效，tab关闭即清除，不同tab间隔离。

localStorage为什么不随tab关闭销毁：持久化到磁盘，浏览器视为用户主动保存的数据，只有 localStorage.clear() 才清除。sessionStorage才是跟tab生命周期绑定的。

SSE的停止方式：前端用 eventSource.close() 或 AbortController.abort()；服务端关闭stream或发送约定结束事件（SkinMe用 data:[DONE]）；断线重连用Last-Event-ID从断点续传。
N6SSE vs WebSocket——选型依据
必问▼
SSE（Server-Sent Events）：单向，服务器→客户端，基于HTTP，自动重连，实现简单。适合AI流式输出、消息推送等单向场景。
WebSocket：全双工，双向通信，需要独立握手协议，适合实时协作、游戏、需要客户端主动发消息的场景。

SkinMe用SSE：用户发一次请求，Gemini持续推token，单向推送，SSE更合适。用fetch+ReadableStream实现而非EventSource，方便配合AbortController取消。
Audeate用WebSocket：需要前端发送控制指令（开始/暂停采集），必须双向，所以用WebSocket。高频数据（50-200Hz）用ref缓冲+requestAnimationFrame批量更新，不每帧setState；波形图用Canvas而非SVG，绕过React渲染树，性能好几个数量级；断线重连用exponential backoff（3s→6s→12s，上限30s）。

T1webpack vs vite——为什么vite开发环境快
必问▼
webpack：先把所有文件打包成bundle再启动服务器。项目1000个文件，先处理1000个才让你看页面，项目越大启动越慢（可能几十秒）。

vite：先启动服务器，浏览器请求哪个文件再编译哪个（按需编译）。不需要等全部打包，启动几乎瞬间。底层用esbuild做依赖预构建（Go编写，比webpack快100x），开发环境用浏览器原生ES Module。

生产环境：两者都需要打包。vite生产用rollup打包，webpack用自己。所以"vite快"只是开发环境快，生产差距没那么大。

Webpack优化策略：代码分割（路由懒加载）、Tree Shaking、持久化缓存（filesystem cache）、并行编译（esbuild-loader替换babel-loader）、Bundle分析（webpack-bundle-analyzer找大包）。SkinMe用Next.js内置webpack，动态导入 import dynamic from 'next/dynamic' 做代码分割。

如何证明优化效果是你带来的（归因分析）：单次只上一个优化，观察7天；用A/B测试对比；记录上线时间点排除同期其他变量；选稳定基准指标（构建时间用相同机器测）。

P3MCP是什么——你的真实经验怎么说
必问▼
MCP是什么：Model Context Protocol，Anthropic提出的开放协议，标准化AI模型调用外部工具和数据源的方式。类比USB——以前每个AI应用自己写工具调用逻辑，MCP定了统一标准，工具实现MCP接口后所有模型都能调用。

SkinMe里的MCP思想（注意：没有直接用MCP协议）：每个功能模块（皮肤分析/成分查询/产品推荐）注册成独立工具描述自己能做什么。零LLM关键词路由根据意图分发到对应Agent串行执行——路由→单个Agent→RAG检索→生成回答，是一条链路的进度展示，不是并行多Agent。这和MCP的tool dispatch逻辑本质一样。

面经1百度暑期实习前端一面——CSS/JS/Vue基础真题
真题▼
CSS部分：
rem和em的区别：em是相对父元素字体大小；rem是相对根元素（html）字体大小，更容易做全局缩放，移动端适配常用rem。
盒模型：标准盒模型：width只包含content；IE盒模型（border-box）：width包含content+padding+border。CSS设置box-sizing: border-box更直观，SkinMe里统一用这个。
垂直居中的方式：Flexbox（display:flex; align-items:center; justify-content:center）最常用；Grid（place-items:center）；绝对定位+transform（top:50%; transform:translateY(-50%)）；line-height等于height（只适合单行文字）
flex代表什么：flex是flex-grow、flex-shrink、flex-basis的缩写。flex:1等于flex:1 1 0——允许放大、允许缩小、初始大小为0。
display的值：block（块级独占一行）、inline（行内不能设宽高）、inline-block（行内但能设宽高）、flex、grid、none（隐藏）
JS部分：
==和===的区别：==会做类型转换再比较；===严格相等，类型不同直接false。[] == false是true（[]→""→0，false→0）；[] === []是false（引用不同）。
原型链：每个对象有__proto__指向原型，一直到null。创建没有原型的对象：Object.create(null)
作用域和作用域链：作用域是变量可访问的范围（全局/函数/块级）。找变量时从当前作用域往外层找直到全局，就是作用域链。
垃圾回收：标记清除（主流）——标记所有从根可达的对象，清除未标记的。内存泄漏常见原因：全局变量、定时器未清除、闭包持有外部引用、DOM引用未释放。
闭包：函数记住了自己被创建时的环境。用途：数据私有化、函数工厂、回调函数保留状态。
事件循环：同步代码→微任务（Promise.then）→宏任务（setTimeout）。
let和const区别：let可重新赋值，const不可（但对象属性可改）。两者都有块级作用域，都有暂时性死区（TDZ）。
Vue部分：
v-show和v-if：v-show用CSS display:none隐藏，DOM始终存在；v-if直接移除DOM。频繁切换用v-show，条件很少变用v-if。
vue2和vue3区别：vue3用Composition API；响应式从Object.defineProperty改为Proxy（能监听数组和新增属性）；虚拟DOM重写，Tree-shaking支持。

面经3百度Web前端一面——前端基础深度题
真题▼
script async和defer的区别：
普通script：遇到停止HTML解析，阻塞渲染。
async：下载和解析并行，下载完立即执行，顺序不确定，适合独立脚本（统计）。

面经4百度Web前端一面——Vue scoped深度修改 + 冒泡排序
真题▼
Vue scoped下深度修改第三方组件样式：
scoped给选择器加唯一属性，第三方组件DOM没有这个属性，样式穿透不进去。
Vue3用 :deep(.class) { color: red }，Vue2用 ::v-deep .class。
原理：:deep()告诉编译器，这个选择器后面不加scoped属性限制，允许匹配子组件内元素。
React里对应写法：CSS Modules的 :global(.class)。
手写冒泡排序：
function bubbleSort(arr) {
  for (let i = 0; i < arr.length-1; i++) {
    let swapped = false
    for (let j = 0; j < arr.length-1-i; j++) {
      if (arr[j] > arr[j+1]) { [arr[j],arr[j+1]]=[arr[j+1],arr[j]]; swapped=true }
    }
    if (!swapped) break
  }
  return arr
}
O(n²)最坏，O(n)最好（swapped优化），空间O(1)原地排序。

太死板 → 减少过度约束，给AI更多空间

迭代原则：每次只改一个变量，建评估集（20-50个case），对比前后效果，不要凭感觉。

A4Context Engineering——上下文工程是什么，怎么管理
必问▼
Context Engineering是什么：
不只是写好单条Prompt，而是系统性地设计AI在整个任务过程中能看到什么信息——什么时候注入什么内容、信息怎么组织、怎么在有限的上下文窗口里放最有价值的东西。

核心挑战：上下文窗口是有限的
模型有token上限，不能把所有信息都塞进去。关键是取舍——什么信息值得占用宝贵的上下文位置。

四个重要原则：
— 重要内容放末尾：LLM对上下文中间部分注意力衰减（Lost in the Middle），最重要的指令放最后
— 无关内容比没有更有害：塞进去的无关信息会触发联想，产生幻觉，宁可少给不要乱给
— 历史用摘要替换全文：不要把完整对话历史都放进去，压缩成摘要保留关键信息
— 动态组装：根据当前请求类型动态决定注入哪些内容，不要固定模板

SkinMe里的Context Engineering：
每次请求动态组装四块内容——用户画像+实时天气UV+日历事件+7天对话摘要，注入System Context。UV高时主动注入防晒提醒，是根据当前状态动态决定的，不是固定写死的。

Prompt Engineering vs Context Engineering的区别：
Prompt Engineering是写好"一条指令"；Context Engineering是设计"整个信息架构"——AI在整个对话/任务过程中能看到什么、以什么顺序看到。后者是系统级的设计，前者是语句级的优化。

🗣 嘴上怎么说：
"Context Engineering比Prompt Engineering更上一层——不只是写好指令，而是设计AI在整个任务过程中能看到什么。核心是取舍，窗口有限，要决定什么信息值得放进去。SkinMe里我们根据每次请求动态组装上下文，不是固定模板，UV高的时候自动带入防晒相关内容。"

A1缓存策略——你做了什么性能优化？（JD 必问）
必背▼
🗣 嘴上怎么说：

"缓存是SkinMe做得比较完整的一块。分两层：

第一层·用户context缓存（user_context_cache，实际120秒TTL）
每个用户的完整System Context组装结果缓存——把画像+天气+日历+聊天摘要拼好的那个完整字符串存下来，下次直接用，不重新组装。cache key是 context:{email}:{days}:{include_history}。命中率高的原因是用户短时间内多轮对话context基本不变，重新拉GCS是浪费。

关键设计·带weather data的请求绕过缓存。天气变化快，UV从中午到下午变一次，缓存会失真。所以weather走实时路径，其他走缓存。这个决策体现的是按数据更新频率分层的思路。

第二层·画像原始数据缓存（profile_cache，600秒TTL）
这层缓存的是从GCS读出来的profile.json原始数据（肤质/过敏/目标等字段），不是组装后的context。画像比context更稳定——用户不会频繁改肤质和过敏，所以TTL更长，10分钟。

一致性·主动失效：每次ProfileExtractor更新画像后，主动 invalidate 该用户的所有context缓存——遍历cache key，以 context:{email}: 开头的全删。下次请求拿到的一定是最新profile。这是简化版write-through失效。

线程安全：SimpleCache用Python threading.Lock 包了get/set/invalidate/clear，因为FastAPI是多worker并发的。

设计原则·缓存粒度按数据更新频率分层：
— 画像原始数据（天级更新）→ 10分钟TTL
— Context组装（几分钟稳定）→ 2分钟TTL
— 天气（分钟级变化）→ 不缓存"
追问1：这个缓存方案有什么问题？规模化怎么改？

"四个问题：
1·进程内dict，多Pod不共享。规模化要换Redis——跨Pod共享 + 发布订阅做分布式失效。
2·没做缓存预热。高频用户登录时主动拉一次context进缓存，减少冷启动延迟。
3·没做缓存穿透防护。恶意请求不存在的user_id会直接打GCS。解法：布隆过滤器前置拦截，或空值缓存（TTL短，比如30秒）挡住。
4·TTL是固定的。更精细的是热度自适应TTL——访问越频繁TTL越长；或者做LRU淘汰控制内存上限。

A5反思纠错 vs Fallback——Agent/RAG 各层的纠错范式
必背▼
🗣 先分清两个概念——纠错 vs Fallback：

纠错（Self-correction）= 系统发现错了，再试一次做对——换 query 重查、修正输出、反思记忆。
Fallback（降级）= 系统做不了了，给用户一个退而求其次的安全结果——不是修复，是认怂但不让用户看到空白。

SkinMe 有 Fallback，没有真正的纠错。诚实说。
SkinMe 里的 Fallback（已实现·5 处）：

— 产品匹配 0 条 → 返回洁面/保湿/防晒基础产品（recommendation_agent.py fallback 块）
— 图像处理失败 → 降级到纯文本分析（analysis_agent.py except 块）
— GCS 读画像失败 → 返回空画像不阻塞（user_profile_manager.py）
— 聊天记录写失败 → 返回 False 不崩主流程（chat_logger.py）
— 缓存过期 → 重新拉 GCS 重新组装 context（cache.py）

核心原则：用户永远拿到东西，不看到空白或报错。
纠错范式——分两层（RAG 层 vs Agent 层）：

█ RAG 层的纠错范式：

— Self-RAG：模型在生成时自己判断检索够不够好——需要检索吗？检索相关吗？输出有幻觉吗？根据判断决定重新检索或修正输出。是检索和生成交替的。
→ SkinMe 简化版：RAG 相似度低于阈值不注入 + post-hoc 校验成分是否在 RAG 结果里出现

— 幻觉检测+自纠正：生成完之后校验输出是否被检索结果支撑（Faithfulness检查），发现幻觉后重新检索或规则拦截。
→ SkinMe：AVOID 成分硬过滤就是规则层的自纠正——LLM 想推荐 paraben，规则层直接拦截

█ Agent 层的纠错范式：

— ReAct（单 Agent 推理循环）：Reason→Act→Observe→Reason...。Agent 执行一步后看 Observation 判断结果够不够好，不好就换方向。是在线实时的纠错。
→ SkinMe 不需要：流程确定，不需要 Agent 自主决策"下一步做什么"

— Reflexion（单 Agent 记忆反思）：Agent 执行完整个任务后，回顾"我哪里做错了"，把反思存进记忆，下次遇到类似任务参考。是跨任务的纠错。
→ SkinMe 不需要：每次推荐是独立任务，没有"做完一个推荐反思后下次推荐更好"的需求

— Multi-Agent 辩论（多 Agent 协作校验）：多个 Agent 各出一个答案，互相批评，收敛到更可靠的结果。
→ SkinMe 不需要：成本太高（每个推荐跑3个Agent讨论），确定性打分已经够用

为什么 SkinMe 只做 Fallback 不做纠错：
SkinMe 的问题是结构化的——检索→提取→匹配→回复，流程固定。纠错范式解决的是开放问题——"帮我规划一次旅行"这种需要 Agent 自主判断"我做得对不对"的场景。
引入纠错 = 多一次 LLM 调用 = 多一次幻觉机会 + 多一倍延迟 + 多一倍成本。结构化任务用规则兜底比 LLM 自纠正更可靠。
追问1：那什么场景需要纠错而不是 fallback？

"当 fallback 的'退而求其次'用户不能接受时。
— 护肤推荐没匹配到 → fallback 给基础产品 → 用户勉强能接受
— 金融投资建议生成错误 → fallback 说'请咨询理财顾问' → 用户不接受（他就是来要建议的）
后者需要纠错：检测到问题后换策略重新生成，而不是放弃。
阿里的业务场景（电商/金融）大概率需要纠错——用户来下单，你不能 fallback 说'我不会'。"
追问2：ReAct 和 Reflexion 的区别？

"ReAct 是在线的：做一步看一步，当场纠正。适合单次任务内的多步决策。
Reflexion 是离线的：做完整个任务后反思，记忆存下来，下次用。适合跨任务的持续改进。
类比：ReAct 像考试时检查答案当场改，Reflexion 像考完后总结错题本下次考更好。"
追问3：如果让你给 SkinMe 加纠错，你会加哪种？

"加Self-RAG 的简化版——在生成成分推荐后，用一个轻量判断'RAG 召回的文献是否足够覆盖这个皮肤问题'。
— 覆盖够 → 正常输出
— 覆盖不够（比如检索相关性分数 < 阈值的比例 > 50%）→ 换 query 重查一次
— 重查还不够 → 降级到'建议咨询皮肤科医生'
这样在 fallback 之前多了一次'尝试做对'的机会，但不会无限循环（最多重查一次），延迟可控。"
不要说的话：
— "SkinMe 有自纠正能力"（只有 fallback，没有真正的纠错）
— "Fallback 就是纠错"（不一样——纠错是重试做对，fallback 是放弃给替代品）
— 把 ReAct / Reflexion / Self-RAG 混在一起说——分清是 RAG 层还是 Agent 层

— MCP是'给AI连接外部能力'：工具调用协议，让AI能真正执行操作（查数据库、调API、读文件系统）。

追问：Agent编排你怎么理解？

"编排的核心是状态管理+流程控制。三个层次：

加分▼
🗣 嘴上怎么说：

A12RAG 召回质量优化四层（JD 明说）

🗣 嘴上怎么说：

"RAG召回优化分四层：

第1层·Chunk策略。
DermNet文献用 chunk_size=512, overlap=100。chunk太大会稀释relevance——一篇文章里既讲维A酸又讲烟酰胺，检索'维A酸'时整篇被召回但实际有用的只有两段。chunk太小会丢语义——一个推荐理由可能跨段落。512是皮肤科文档的sweet spot，一个完整的'皮肤问题+成因+成分建议'刚好在512 token内。overlap=100防止关键信息被切割在chunk边界。

第2层·Query改写。
不直接把用户原话查RAG——用户说'我脸上长痘怎么办'，文献里不会有这种口语。代码里重写成 'What are the most effective ingredients for treating {condition}?'，更接近文献的语言风格，召回质量明显更好。这是简化版的query rewriting。更复杂的做法是HyDE（让LLM先生成假设答案做embedding检索），SkinMe没用。

第3层·top-k调参。
现在用 similarity_top_k=10。少了（5）召回不全，很多有用的边缘成分查不到；多了（20）引入噪声，不相关的文献占context位置，反而触发LLM联想幻觉。做过5/10/20的对比，10是sweet spot。

第4层·后处理。
SkinMe这层做得较弱，只做了程序校验（推荐的每个成分必须在RAG返回的文献里真的出现过，没出现的过滤掉）。更完善的做法是上reranker——检索出20条用cross-encoder重排选前5，比纯向量相似度更准。SkinMe没上是因为Gemini自己对10条上下文的attention已经够用。"

A15幂等性设计——金融/电商场景必考
必考▼
🗣 嘴上怎么说：

"幂等性的意思是：同一个操作执行一次和执行多次，结果完全一样。网络超时重试时，不能因为重试就扣两次款。

标准解法三步：
1·客户端生成全局唯一 order_id（UUID），随请求发给服务端。
2·服务端执行前先查：这个 order_id 是否已经处理过？已处理且成功 → 直接返回成功，不重复执行。
3·数据库给 order_id 加唯一索引：即使查询和写入之间有并发窗口，唯一索引保证只有一个 INSERT 成功，第二个会唯一键冲突，捕获异常返回原结果。

公式：幂等性 = 唯一标识(order_id) + 执行前查状态 + 数据库唯一索引(防并发)

SkinMe 里的体现：
— 读操作天然幂等（查RAG、查产品），重试安全
— 写操作（ChatLogger.log_message、ProfileExtractor 写GCS）目前没做幂等——是已知缺陷，规模化要加 request_id
— 规划中：每个请求加 request_id，写入前查 GCS 该 ID 是否已存在"
追问：GET 天然幂等，POST/PUT/DELETE 呢？

"GET：天然幂等，查多少次结果一样。
PUT：语义上幂等——'设置余额为100'执行两次还是100。但如果实现成'余额+100'就不幂等了，这是设计错误。
DELETE：第一次删成功返回200，第二次删不存在返回404。严格说HTTP状态码不同，但业务层面效果幂等（东西确实没了）。
POST：天然不幂等（创建资源），必须用 order_id + 唯一索引保护。"
追问：分布式场景怎么保证幂等？

"单机用数据库唯一索引就够。分布式要用Redis SETNX做分布式幂等锁——SETNX order_id 1 EX 3600，设置成功说明第一次处理，设置失败说明已在处理。配合 TTL 防止死锁。
更严格的场景用TCC事务：Try冻结资金→Confirm扣款→Cancel解冻。每步都有明确的前进/回滚路径。"
腾讯面经教训：面试官问"重试有没有安全风险"，你答了"有"但没说清解法。直接背公式：唯一标识+查状态+唯一索引。
A16TLS 密钥保护——为什么中间人截不走密钥
必考▼
🗣 嘴上怎么说：

"TLS握手的核心：

1·客户端Hello（算法列表+随机数A）→ 2·服务器Hello + 证书 + 随机数B → 3·客户端验证证书合法（CA签名/域名/过期），生成随机数C，用服务器公钥加密C发过去 → 4·双方用A+B+C生成同一个对称密钥 → 5·6·Finished

为什么中间人截不走（核心那句话）：
随机数C在传输前用服务器公钥加密。全网只有服务器持有对应私钥才能解开。中间人截到的是密文，没有私钥解不开——即使截到了也没用。

为什么不全程用非对称加密：
RSA加解密比AES慢1000倍量级。所以非对称只用于握手阶段传递那一个随机数C（几十字节），之后用对称加密AES传大量数据。这是性能和安全的工程折中。

证书怎么验证合法：
证书里有服务器公钥 + CA的数字签名。客户端用预装的CA公钥验签——签名合法说明证书没被篡改，公钥确实是这个域名的服务器的。证书链：服务器证书→中间CA→根CA，一层层验到预装的根证书。"
腾讯面经教训：你说到了三个随机数，但没说"C用公钥加密"这句核心话。这句话下次必须说出来。
A17Chunk 为什么不整篇放——噪声+语义聚焦两个原因
必考▼
🗣 嘴上怎么说（两个原因都说）：

"第一个原因·噪声干扰（面试官最想听的）：
整篇论文包含大量无关内容——作者介绍、参考文献、致谢、摘要重复、实验设备清单。这些和具体皮肤科知识无关，但会被编码进向量里成为噪声，降低检索精度。chunk切片能把这些无关段落和有用段落分开，检索时只召回有用的。

第二个原因·语义聚焦：
一篇论文可能讲5种皮肤问题的成分。如果整篇放进一个向量，这个向量代表的是'5种问题的混合语义'，和任何一种具体问题都不够相似。切成chunk后，每个chunk只包含1-2个知识点，向量更精确地表达单个知识点，检索命中率更高。

SkinMe 的参数：chunk_size=512, overlap=100
— 512：一个完整的'皮肤问题+成因+成分建议'刚好放进512 token
— overlap=100：防止关键信息被切割在chunk边界——一句话的前半段在chunk A末尾，后半段在chunk B开头，overlap保证两个chunk都包含完整句子
— 做过语义分块实验：效果略好但计算成本高，短文本场景优势不明显，保留固定窗口"
腾讯面经教训：你只说了"向量被稀释"（语义聚焦），没说"噪声干扰"。两个原因都要说，噪声那个更关键。

A19Pydantic vs PEP8——不要混淆
注意▼
两个完全不同的东西：

PEP8：Python代码风格规范——缩进4空格、变量名snake_case、行长度不超79/120、import顺序。用Ruff/Black/Flake8自动检查和格式化。SkinMe的CI里用Ruff做lint。是代码怎么写好看的问题。

Pydantic：Python的数据验证库——定义数据模型的schema，运行时自动校验类型和格式。在AI应用里用于Agent输出的结构化校验——LLM返回的JSON如果不符合schema，Pydantic直接报ValidationError，可以触发re-prompt或fallback。是数据对不对的问题。

SkinMe里的用法：
— PEP8：CI里Ruff lint（ruff check . + ruff format --check .）
— Pydantic：FastAPI的请求/响应模型用了Pydantic自动校验（main.py里的endpoint参数）；但Agent输出校验用的是Gemini的response_schema而不是Pydantic——效果类似但实现不同

面试官可能追问的角度："你用Pydantic做什么？"
→ "FastAPI天然集成Pydantic，请求参数自动校验类型——比如/chat的message字段是string，不传或传错类型直接422。Agent输出校验我们用了Gemini的response_schema，本质和Pydantic一样是schema约束，但在API层面而不是应用层面。""
腾讯面经教训：你把Pydantic说成了PEP8。PEP8管代码风格，Pydantic管数据校验。两回事。

A21Self-RAG（自我反思 RAG）——不只是检索+生成
技术深度▼
🗣 嘴上怎么说：

"普通RAG是一次性的：检索→注入→生成，结束。
Self-RAG在生成过程中加入反思步骤——模型在每个生成片段时同时判断三件事：
— 这个片段需要检索吗？（有些问题不需要外部知识）
— 检索到的内容和问题相关吗？（低相关就丢弃，不注入）
— 我的输出有幻觉吗？（生成的内容有没有超出检索结果的范围）

根据反思结果决定：继续生成 / 重新检索（换query或换策略）/ 修正输出。是检索和生成交替进行的，不是检索完一次就不管了。"
SkinMe 里的简化版 Self-RAG：

"SkinMe 没有完整 Self-RAG，但有两个点是同样思路：
— RAG 相似度阈值：检索结果相似度低于阈值不注入——这就是'判断检索到的内容是否相关'这一步
— Post-hoc 校验：推荐的成分必须在RAG结果里真出现过，没出现过的过滤——这是'判断输出有没有幻觉'的简化版
— 没做的：模型自己动态判断'要不要检索'、'要不要换query重检索'——这需要专门训练的反思 token（Self-RAG 论文用了 reflection tokens），SkinMe 用规则替代了。

完整 Self-RAG 的价值在长尾问题——标准检索覆盖不到的边缘case，模型自己意识到'我不够确定'然后主动换策略重查。SkinMe 目前的做法是遇到不确定就'建议咨询皮肤科医生'，是更保守但更安全的降级。"
追问：Self-RAG 论文的核心机制？

"在训练阶段给模型加入了四种特殊 token（reflection tokens）：
— Retrieve：决定是否需要检索
— IsRel：判断检索结果是否相关
— IsSup：判断生成内容是否被检索结果支撑
— IsUse：判断生成结果是否对用户有用
模型在生成每段输出时同时输出这些 token，训练数据用 GPT-4 标注。推理时根据这些 token 的值决定是否重新检索或修正。

我没从零实现过这个，但理解机制。SkinMe 用规则实现了类似效果的子集。"
不要说的话：
— "SkinMe 实现了 Self-RAG"（没实现完整版，只有思路类似的规则）
— "Self-RAG 和普通 RAG 没区别"（区别很大，Self-RAG 是动态的）
A22幻觉检测与自纠正——检测到幻觉后怎么办
技术深度▼
🗣 嘴上怎么说：

"幻觉控制分两步：检测和纠正。很多系统只做了检测没做纠正——发现幻觉了直接报错，但没修复。

检测方法（三种）：
— Faithfulness 检查：比较生成内容和检索source是否一致——生成内容里有没有source里不存在的事实？用 NLI（自然语言推理）模型判断'生成内容能否从source推导出来'。RAGAS 的 faithfulness 指标就是这个。
— 自一致性检查：同一个问题跑多次，如果答案差异大（temperature>0 时），说明模型不确定，幻觉概率高。稳定的答案更可信。
— 规则校验：SkinMe 的做法——推荐的成分必须在RAG结果里出现过，没出现过 = 幻觉，直接过滤。最简单但最可靠。

纠正方法（三种）：
— 重新检索：换 query 或换检索策略再查一次，用新的 context 重新生成。HyDE（假设文档嵌入）就是改进 query 的方式。
— 规则层拦截：SkinMe 的做法——LLM 要推荐 AVOID 成分时，规则层直接拦截，不让输出。这是硬编码的自纠正，不依赖 LLM 自觉。
— 降级：检测到低置信度直接说'建议咨询皮肤科医生'。宁可不答也不答错——护肤场景的安全底线。

SkinMe 的五层防御本质上就是'检测+纠正'的工程化：
RAG锚定（防止生成超出知识范围）→ 结构化输出（检测格式异常）→ 低temperature（减少随机性）→ Post-hoc校验（检测成分是否在RAG里）→ AVOID硬过滤（规则层纠正）。"
追问：生产中大规模做幻觉检测的成本问题？

"幻觉检测本身要调 NLI 模型或 LLM 做判断——等于每个回复多一次模型调用，成本翻倍，延迟增加。
折中方案：
— 高风险场景全量检测（金融/医疗推荐），低风险场景（闲聊）跳过
— 采样检测：线上 10% 请求做幻觉检测，发现幻觉率上升才全量开启
— 轻量规则先过滤：SkinMe 的做法——先用成本几乎零的规则校验（成分是否在RAG里出现），过不了再用重量级检测
分级处理，不是全量上重检测。"
不要说的话：
— "我们做了 NLI 幻觉检测"（没做，用的是规则校验）
— "幻觉可以完全消除"（不可能，只能控制概率和后果）
A23自适应路由（Adaptive Routing）——不同问题走不同链路
技术深度▼
🗣 嘴上怎么说：

"自适应路由的核心：不同问题适合不同检索/处理策略，不要所有请求走同一条链路。

SkinMe 的 RoutingAgent 就是自适应路由的简化实现：
classify_intent_fast 根据关键词把用户意图分成 5 种，每种走不同链路：
— analysis_only：只走 Vertex RAG 做皮肤分析，不推荐产品
— recommendation：走 RAG 分析 + EWG 产品匹配全链路
— both：带图片上传，走分析 + 推荐
— image_history：走图像历史对比（Gemini 多图分析）
— none：纯闲聊，不调任何 Agent，直接 Gemini 聊天

为什么 none 路由很重要：'你好'这种问候不调 RAG 不查产品——节省了 Gemini API 调用 + RAG 检索 + 产品匹配的全部成本和延迟。大量用户第一句话就是'你好'，如果全走 Agent 链路是巨大浪费。

更复杂的自适应路由（SkinMe 没做但了解）：
— 按 query 置信度动态选检索策略：高置信度（明确的成分词）→ BM25 精确匹配更好；低置信度（模糊的皮肤描述）→ Dense 向量检索更好；极低置信度 → 直接 LLM 兜底不查 RAG
— LLM 做路由器：让 LLM 判断走哪条链路——灵活但增加延迟和不确定性。SkinMe 用关键词规则而不是 LLM 做路由就是为了确定性和零延迟"
追问：为什么路由不用 LLM？用了会怎样？

"三个理由：
— 延迟：LLM 路由多一次 API 调用，~200-500ms。路由是每次请求第一步，延迟直接加在用户等待时间上。关键词匹配几乎零延迟。
— 确定性：路由错了影响整条链路——把'推荐产品'路由成'纯闲聊'，用户什么都拿不到。关键词规则的行为 100% 可预测可审计。
— 成本：路由判断不需要语义理解能力。'recommend product' 包含 'product' 关键词就该走推荐——用 LLM 做这种判断是大炮打蚊子。

什么时候该用 LLM 路由：意图分类边界模糊、有大量长尾 intent、关键词覆盖不了的场景。SkinMe 只有 5 种 intent，关键词覆盖 95%+，不需要 LLM。"
不要说的话：
— "路由就是 if-else"（这是技术事实但说出来太轻描淡写——说"确定性关键词匹配"更专业）
— "LLM 路由不好"（不是不好，是不适合这个场景）
对应代码：routing_agent.py 的 classify_intent_fast（5种intent + 关键词列表）+ route_and_process（根据intent调用不同Agent组合）
A24Agent 状态机设计——LangGraph 和更好的编排
技术深度▼
🗣 嘴上怎么说：

"把多 Agent 系统看成状态机：每个 Agent 是一个状态，状态之间的转移有条件。

SkinMe 现在的'隐式状态机'：
route_and_process 函数里的逻辑其实就是一个状态机，只是用 if-else 写死了：
— 状态1（路由）→ intent=analysis → 状态2（分析）
— 状态2（分析）→ 分析完成 → 状态3（推荐）或直接返回
— 状态3（推荐）→ 有产品 → 返回结果；无产品 → 走fallback
— 任何状态出错 → 返回错误/降级

问题：状态转移逻辑和Agent执行逻辑混在一起。新增一个状态（比如'人工审核'节点、'用户确认'节点）要改 route_and_process 里的所有分支——不符合开闭原则。

LangGraph 怎么做得更好：
LangGraph 把状态机显式化：
— 节点：每个 Agent/操作是一个节点函数
— 边：节点之间的连接，可以是无条件（A做完一定到B）或条件（根据A的输出决定到B还是C）
— 状态：一个 TypedDict 在图里传递，每个节点读写状态
— 特殊节点：END（结束）、human_in_the_loop（等待人工确认）

好处：
— 新增节点不改已有节点（只加一条边）
— 循环和回退天然支持——'检索不够好就回到检索节点换query重来'用一条边就搞定
— 可视化状态图，调试时能看到请求走了哪条路径

SkinMe 如果重新设计：
把 route_and_process 的 if-else 改成 LangGraph 状态图。增加两个新节点：
— 质量检查节点：分析完成后检查RAG召回质量，太差就回到检索节点换query
— 降级节点：任何节点连续失败N次自动转入降级状态，返回安全回复
这两个在 if-else 里实现会让代码变成意大利面条，LangGraph 里就是两条边。"
追问：状态机和普通 pipeline 的核心区别？

"Pipeline：A→B→C→D，单向、确定性、没有分支没有回退。SkinMe现在是这个。
状态机：A→B，B判断结果→好走C→差走回A重试→极差走D降级。有条件分支和循环。
核心区别是能不能走回头路。Pipeline不能，状态机能。
什么时候需要走回头路：检索质量不够需要换query重查、用户输入歧义需要追问、工具调用失败需要换工具——这些都是开放任务才有的需求，SkinMe的确定性流程不太需要，但阿里的业务场景可能需要。"
不要说的话：
— "SkinMe用了LangGraph"（没用，用的是手写if-else）
— "状态机比pipeline好"（不是好坏问题，是匹配问题——简单流程pipeline更好）
— "我实现过LangGraph状态图"（没实现过，了解概念+看过文档/教程）
A25Context Engineering——上下文工程完整版（JD 明说）
必背▼
🗣 嘴上怎么说：

"Context Engineering 比 Prompt Engineering 更上一层——不只是写好一条指令，而是系统性设计 AI 在整个任务过程中能看到什么信息、什么顺序看到、什么时候该注入什么时候不注入。

Prompt Engineering 是语句级优化——这一句 prompt 怎么写效果最好。
Context Engineering 是系统级设计——整个上下文窗口怎么组织，哪些信息值得占位置，哪些应该去掉。"
四个核心原则（必背）：

1·重要内容放末尾（Lost in the Middle）
LLM 对上下文中间部分注意力衰减——Stanford 论文 Lost in the Middle 证实了这一点。最重要的指令和约束放在 System Context 末尾。SkinMe 里用户画像和安全约束（AVOID 成分、SPF 强制推荐）放在 System Instruction 末尾，不放中间。

2·无关内容比没有更有害
塞进去的无关信息不是"没用"——它会触发 LLM 联想，产生幻觉。比如在护肤查询里注入用户的日历事件"下周有会议"，LLM 可能联想"会议压力导致皮肤问题"编出一段话。宁可少给不要乱给。

3·历史用摘要替换全文
完整对话历史 token 成本爆炸，而且触发 Lost in the Middle。SkinMe 用 chat_logger.get_conversation_summary 把 7 天历史压成约 20 词摘要（关键词规则抽取：过敏/敏感/皮肤问题/图片上传），注入 System Context。不是 LLM 摘要——规则抽取零延迟零成本。

4·动态组装，不固定注入
根据当前请求类型决定注入什么，不用固定模板。SkinMe 的 UserContextRetriever.get_smart_context 实现了这个——
SkinMe 的 Context Engineering 代码级实现：

get_smart_context 根据用户意图动态筛选注入哪些 context：
— 问产品/成分（包含 product/ingredient/recommend/buy/routine 等关键词）→ 注入用户画像（肤质、过敏、敏感成分）——安全关键，推荐含过敏成分的产品是事故
— 有日历事件 → 注入日历（CalendarManager 检测 outdoor 关键词 → 高 UV 天提醒防晒）
— 有 weather data → 注入天气（UV 指数、温湿度）
— 有图片上传 → 注入完整 context（全量画像+天气+日历+历史）
— 简单问候（hi/hello/thanks 且 ≤3 词）→ 不注入任何 context，节省 token

为什么简单问候不注入：用户说"你好"时注入 800 token 的画像+天气+历史完全浪费——每次注入都有成本（token 费 + 延迟 + 稀释有效信息），要按需。

组装后的 System Context 格式：
=== SYSTEM CONTEXT - IMPORTANT: Use this factual information ===
USER PROFILE: Skin type: oily | Allergies: fragrance
Current Weather: Temperature 32°C, UV Index 8
📅 TODAY: Outdoor tennis [⚠️ Sun protection needed!]
CHAT HISTORY: concern: acne; uploaded skin image
=== END CONTEXT ===

然后拼上 User query: {用户的实际问题}
追问1：缓存和 Context Engineering 的关系？

"Context 组装本身有成本——要读 GCS 拿画像、扫历史抽关键词、调天气 API。SkinMe 用 user_context_cache（120 秒 TTL）缓存组装好的 context，同一用户多轮对话不重复组装。
但带 weather data 的请求绕过缓存——天气变化快，UV 从中午到下午会变，缓存会失真。
这是Context Engineering 和性能优化的交叉点：context 的'新鲜度'和'组装成本'要平衡。"
追问2：如果上下文窗口不够放所有信息怎么办？

"四个策略按优先级：
1·压缩：历史用摘要替全文（SkinMe 已做）
2·筛选：只注入和当前 query 相关的 context（get_smart_context 已做）
3·分层：最重要的放 system prompt（高注意力），次要的放 user turn 前文
4·外挂记忆：长期画像存 GCS/向量库，用 RAG 按需召回而不是全量注入
SkinMe 做了前三个。第四个是规模化后要做的——画像字段多了之后不可能全塞进 context。"
追问3：Prompt Engineering vs Context Engineering 你怎么区分？

"Prompt Engineering是'这一句话怎么写'——角色定义、任务说明、输出格式、few-shot、约束。对应 SkinMe analysis_agent.py 里的 analysis_prompt。
Context Engineering是'整个窗口怎么组织'——什么信息注入、什么顺序、什么时候注入什么时候不注入。对应 user_context_retriever.py 的 get_smart_context。
前者是单条指令的优化，后者是信息架构的设计。两者配合才完整——好的 prompt 如果被错误的 context 包围，效果也会很差。"
不要说的话：
— "Context Engineering 就是 Prompt Engineering"（不一样，层次不同）
— "我们把所有信息都塞进 context"（这正是 CE 要避免的）
— "摘要是用 LLM 生成的"（SkinMe 用的是关键词规则抽取，零成本）
对应代码：user_context_retriever.py（get_smart_context 第130行·意图关键词列表·skip_keywords·weather 绕过缓存判断·=== SYSTEM CONTEXT === 格式化）+ chat_logger.py（get_conversation_summary·关键词规则·~20词摘要）+ cache.py（user_context_cache 120秒 TTL）+ runner.py（System Instruction 末尾放安全约束）
